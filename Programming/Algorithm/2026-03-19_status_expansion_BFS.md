# 상태 확장 BFS

같은 위치라도 상태(state)에 따라 행동이 달라질 때,\
(위치 + 상태)를 하나의 노드로 보고 탐색을 하는 BFS

---

## 1. 언제 사용해야하지?

다음과 같은 질문에 대해 YES라면 고려해본다.

- 같은 칸에 도착했는데 상황에 따라 이동 가능 여부가 달라지는가?

#### 대표상황

- 벽을 이미 부쉈는지
- 점프를 몇 번 남겼는지
- 열쇠를 가지고 있는지

---

## 2. 일반 BFS vs 상태 BFS

| 구분      | 일반 BFS        | 상태 BFS               |
| ------- | ------------- | -------------------- |
| 노드      | (x, y)        | (x, y, state)        |
| visited | visited[x][y] | visited[x][y][state] |
| 핵심      | 위치만 중요        | 상태까지 중요              |
| 난이도     | 낮음            | 중~상                  |

---

## 3. 자주 하는 실수

### 1) `visited`를 2차원으로 둔다.

```python
visited[x][y]
```

=> 상태 구분 불가

### 2) 상태를 `grid` 원본 배열에 반영

```python
gird[nx][ny] = 0
```

=> 모든 경로에 영향을 준다. => 상태는 큐/`visited`에서 관리한다.

#### Ex. boj2206_벽 부수고 이동하기 문제

해당 문제를 처음 풀 때 `visited`를 다음과 같이 선언했었다.

```python
visited = [[False] * M for _ in range(N)]
```

상태를 고려하지 않으면 다음과 같은 문제가 발생한다:

- 초기 상태 `(0, 0)`에서 시작
- 이후 가능한 이동:
    - `(0, 1)` => 빈칸
    - `(1, 0)` => 벽(부수고 이동 가능)
    
결국 큐 안에는 두 가지가 들어가며, 방문 처리는 다음과 같이 된다.
```python
(0, 1, 2, 0) # 벽 안 부숨
(1, 0, 2, 1) # 벽 이미 부숨

visited[0][1] = True
visited[1][0] = True
```
- 문제는 `(1, 1)` 지점에서 발생한다.
    - `(1, 1, 0)` => 아직 벽을 부술 수 있음
    - `(1, 1, 1)` => 더 이상 벽 부수기 불가
    - 하지만 코드에서는 `visited[1][1]`로 **하나로 처리되어 버린다.**
    
**즉, 먼저 도착한 상태가 이후 모든 경우를 막아버린다.**
